# 06-01 Plan: Google OAuth and Token Storage Boundary

## Objective

Persist the Google provider token boundary in the API/Postgres layer and expose a safe Calendar connection state to the web app.

## Tasks

1. Add a DB repository for `calendar_connections` and `calendar_sync_events`.
2. Add an API calendar service that maps DB rows to public API types and keeps memory fallback for test/dev mode.
3. Encrypt provider access/refresh tokens before storing them.
4. Add an internal API endpoint for the Next.js OAuth callback to save provider tokens after Supabase exchange.
5. Update Calendar route handlers and lesson create/update hooks to use the service instead of `memoryStore`.
6. Update smoke tests for authorization-required connection state and successful provider-token save.

## Success Criteria

- `/calendar/connection` reads DB-backed state when Postgres is configured.
- `/calendar/provider-tokens` stores encrypted tokens and returns `tokenAvailable: true` without exposing secrets.
- Lesson create/update creates DB-backed sync records when Postgres is configured.
- Existing memory fallback tests keep passing.

## Out Of Scope

- Calling Google Calendar APIs.
- Refreshing expired access tokens.
- Listing available calendars from Google.
